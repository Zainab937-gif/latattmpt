Most code you write today will not exist in five years. The startup will pivot. The product will die. The framework will be abandoned.

This used to bother me. Now I think it's a feature, not a bug.

## The Disposable Mindset

When you know something won't last forever, you stop over-engineering it. You stop building for imaginary future requirements that never materialize.

You build for **now**.

## When It Actually Matters

But some things should last. Infrastructure. APIs that other people depend on. Core business logic.

For these things, the bar is different:

1. **Clear interfaces** over clever internals
2. **Boring technology** over exciting new frameworks
3. **Documentation** written like you'll forget everything in 6 months (you will)

## The Rule I Use

Ask yourself: *"Will someone else curse my name because of this decision in 2 years?"*

If yes — slow down.
If no — ship it.

That's it. That's the whole rule.
